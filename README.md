# Kirkalab

Kirkalab backend: a FastAPI service providing user management with JWT authentication and role-based access control (RBAC). The service uses PostgreSQL for persistence, SQLAlchemy 2 as the ORM, Alembic for schema migrations, and ships with Docker and Docker Compose for containerized deployment.

## Features

- JWT-based authentication (`/auth/login`) using HTTP Bearer tokens.
- Role-based access control: admin-only endpoints guarded by an `is_admin` check, plus an active-user (`is_active`) check on every authenticated request.
- User management CRUD: list, create, read, update (PATCH), and delete users.
- First-admin bootstrap on startup, configurable via environment variables (idempotent).
- Database schema managed entirely by Alembic migrations (no `create_all` at runtime).
- Containerized with Docker; the entrypoint applies migrations before the app starts.
- Health check endpoint at `/health`.

## Tech stack

- **Language:** Python 3.12
- **Framework:** FastAPI + Uvicorn
- **Database:** PostgreSQL 16
- **ORM:** SQLAlchemy 2.0 (with psycopg)
- **Migrations:** Alembic
- **Auth:** python-jose (JWT), passlib + bcrypt (password hashing)
- **Tooling:** pytest, httpx, ruff

## Configuration

Configuration is read from environment variables. Copy `.env.example` to `.env` and fill in real values. **Never commit the real `.env` file** — it is git-ignored.

| Variable | Description | Example |
| --- | --- | --- |
| `APP_NAME` | Application name shown in API docs | `kirkalab` |
| `ENVIRONMENT` | Deployment environment | `development` |
| `DEBUG` | Enable debug mode | `false` |
| `DATABASE_URL` | SQLAlchemy database URL | `postgresql+psycopg://user:password@localhost:5432/kirkalab` |
| `SECRET_KEY` | Secret used to sign JWTs (use a long random value) | `openssl rand -hex 32` |
| `ACCESS_TOKEN_EXPIRE_MINUTES` | JWT lifetime in minutes | `60` |
| `ALGORITHM` | JWT signing algorithm | `HS256` |
| `FIRST_ADMIN_EMAIL` | Optional bootstrap admin email | `admin@example.com` |
| `FIRST_ADMIN_HANDLE` | Optional bootstrap admin handle | `admin` |
| `FIRST_ADMIN_PASSWORD` | Optional bootstrap admin password | `change-me` |

If all three `FIRST_ADMIN_*` values are set, an admin user is created on startup. The step is idempotent: it is skipped if a user with `FIRST_ADMIN_EMAIL` already exists.

## Quick start with Docker Compose

The easiest way to run the full stack (API + PostgreSQL) is Docker Compose.

1. Create your environment file:

```bash
cp .env.example .env
# edit .env and set SECRET_KEY, FIRST_ADMIN_*, etc.
```

2. Build and start the services:

```bash
docker compose up --build
```

Compose starts two services:

- `db` — PostgreSQL 16 with a persistent `postgres_data` volume and a health check.
- `app` — the API, which waits for the database to become healthy, applies Alembic migrations, then starts Uvicorn on port 8000.

`DATABASE_URL` is overridden in Compose to point at the `db` service host; all other secrets are loaded from your `.env` file.

The API is then available at `http://localhost:8000`, with interactive docs at `http://localhost:8000/docs`.

## Local development (without Docker)

Requires Python 3.12 and a running PostgreSQL instance.

```bash
python -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt

cp .env.example .env   # set DATABASE_URL to your local Postgres

alembic upgrade head   # apply migrations
uvicorn app.main:app --reload
```

## Database migrations

The schema is managed exclusively by Alembic; the application never calls `create_all` at runtime. The database URL is loaded from the application settings in `alembic/env.py`.

```bash
# apply all migrations
alembic upgrade head

# create a new autogenerated revision after changing models
alembic revision --autogenerate -m "describe change"

# roll back the most recent migration
alembic downgrade -1
```

In Docker, `docker-entrypoint.sh` runs `alembic upgrade head` automatically before starting the server.

## API

All versioned routes are served under the `/api/v1` prefix. Interactive documentation is available at `/docs` (Swagger UI) and `/redoc`.

### Authentication

- `POST /api/v1/auth/login` — exchange email + password for a JWT access token.

Authenticated requests must send the token via the `Authorization: Bearer <token>` header. Inactive users are rejected with `403`.

### Users

All user endpoints except creation require an authenticated **admin** user.

| Method | Path | Description |
| --- | --- | --- |
| `GET` | `/api/v1/users/` | List users (supports `skip` and `limit` query params) |
| `POST` | `/api/v1/users/` | Create a user (returns `201 Created`) |
| `GET` | `/api/v1/users/{user_id}` | Retrieve a single user |
| `PATCH` | `/api/v1/users/{user_id}` | Update a user |
| `DELETE` | `/api/v1/users/{user_id}` | Delete a user (returns `204 No Content`) |

### Health

- `GET /health` — returns `{"status": "ok"}`.

## Data model

The `users` table contains:

- `id` — primary key
- `email` — unique, indexed
- `handle` — unique, indexed
- `hashed_password` — bcrypt hash
- `is_active` — boolean, defaults to `true`
- `is_admin` — boolean, defaults to `false`
- `created_at` — timezone-aware timestamp, set by the database
